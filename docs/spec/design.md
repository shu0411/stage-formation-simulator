## 1. 仕様設計

### 1.1 対象範囲

#### 対象

- ステージを上から見た 2D エディターでのフォーメーション編集
  - メンバーの追加・削除
  - ドラッグによる立ち位置変更
  - メンバー名の編集
- フォーメーションのリアルタイム 3D 表示（簡易人物モデル・メンバー名表示。
  客席中央からの固定視点）
- フォーメーションデータの JSON エクスポート・インポート
- LocalStorage への自動保存と起動時の復元

#### 対象外

concept.md の「MVP対象外（将来拡張）」に挙げられた以下は今回作らない。

- 時系列フォーメーション、フォーメーションアニメーション、動線表示
- AI 機能（立ち位置表画像の解析、フォーメーション評価、重なり検出、バランス分析）
- メンバーの向き・身長設定、照明、エフェクト、客席位置の自由指定
- ステージサイズ変更、会場プリセット、花道対応
- セットリスト管理、複数フォーメーション管理、クラウド保存

また、本仕様の検討過程で次も対象外と決めた。

- 2D エディターでのグリッドへのスナップ（吸着）— 将来の追加開発要素とする
- カメラ視点の切り替え（客席左・客席右・自由視点を含む）— MVP は客席中央からの
  固定視点のみとする（1.5 3D プレビュー）
- タッチ操作への最適化（タブレット・スマートフォンでの表示自体は行う。1.6）

### 1.2 想定ユーザーと利用場面

- **想定ユーザー**: UNIDOL チーム、アイドルグループ、ダンスチーム、振付担当、
  ライブ演出担当
- **利用場面**: ライブ・発表の演出検討時に、フォーメーションを 2D で作成し、
  客席からどう見えるか（メンバーの重なり、全体のバランス）を 3D で確認する

### 1.3 機能概要

ステージフォーメーションを 2D エディターで作成・編集し、その内容をリアルタイムに
3D 空間へ反映して客席視点から確認できる Web アプリケーション。

**アプリ全体の入力・出力（外部との境界）**

- 入力: 画面上の GUI 操作、フォーメーション JSON ファイル（インポート）
- 出力: フォーメーション JSON ファイル（エクスポート）
- 保存先: ブラウザの LocalStorage（保存ボタンによる手動保存・起動時復元）

**機能一覧**（詳細は 1.5）

1. 2D エディター表示（サムネイル・編集ポップアップ）
2. メンバー追加
3. メンバー削除
4. 立ち位置変更（ドラッグ・数値入力）
5. メンバー名編集
6. 3D プレビュー
7. JSON エクスポート
8. JSON インポート
9. 保存・復元

### 1.4 画面・操作フロー

画面は 1 画面構成とし、画面の大部分を 3D ビューが占める。2D エディターは
画面左上のサムネイルと、それをクリックして開く編集ポップアップで構成する。

画面の構成要素:

- 3D ビュー（画面の大部分。客席中央からの固定視点でフォーメーションを 3D 表示する）
- 2D サムネイル（画面左上。現在のフォーメーションの俯瞰図。
  詳細は 1.5 の 2D エディター表示）
- 2D 編集ポップアップ（サムネイルのクリックで開く拡大された俯瞰図。
  メンバーの追加・削除・立ち位置変更（ドラッグまたは数値入力）・名前編集を行う）
- 保存ボタン（LocalStorage への保存。1.5 保存・復元）
- JSON エクスポート・インポートの操作（1.5）

利用の流れ:

```text
アプリを開く
    ↓
LocalStorage に保存データがあれば復元、なければ空のステージ（1.5 保存・復元）
    ↓
サムネイルをクリックして 2D 編集ポップアップを開く（1.5 2D エディター表示）
    ↓
ポップアップ内で編集（追加・削除・立ち位置変更（ドラッグ・数値入力）・名前編集）
    ↓ 編集のたびに
3D ビュー・サムネイルへリアルタイム反映（1.6）
    ↓
ポップアップを閉じ、保存ボタンで LocalStorage へ保存（1.5 保存・復元）
    ↓ 任意のタイミングで
JSON エクスポート / インポート（1.5）
```

### 1.5 機能仕様

#### 2D エディター表示（サムネイル・編集ポップアップ）

- **入力**: サムネイルのクリック、ポップアップを閉じる操作
- **正常系の振る舞い**: 画面左上のサムネイルに、現在のフォーメーションの俯瞰図を
  常に表示する。サムネイルをクリックすると、拡大された 2D 編集ポップアップが開く。
  ポップアップにはステージの俯瞰図とグリッド線（1.6）を表示し、この中で
  メンバーの追加・削除・ドラッグ・名前編集を行う。閉じる操作でポップアップを
  閉じる（編集内容は保持される）
- **出力**: サムネイルおよびポップアップの表示
- **エラー・空状態**: メンバーが 0 人のときは、ステージのみの俯瞰図を表示する
- **受け入れ条件**:
  - サムネイルをクリックすると、2D 編集ポップアップが開く
  - ポップアップを閉じると元の画面に戻り、編集した内容は保持されている
  - フォーメーションを編集すると、サムネイルの表示も追従して更新される
  - ポップアップ内のステージにはグリッド線（1.6）が表示される

#### メンバー追加

- **入力**: 追加操作（追加ボタンの押下）
- **正常系の振る舞い**: 新しいメンバーが 1 人、原点（x=0, y=0）に配置され、
  自動的に選択状態になる（1.6 メンバーの選択）。
  **仮定:** 名前の初期値は「メンバー1」「メンバー2」…の連番とする
- **出力**: 追加されたメンバーが 2D エディターと 3D ビューに表示され、選択中
  として区別される
- **エラー・空状態**: なし（メンバー 0 人の状態からでも追加できる）
- **受け入れ条件**:
  - メンバーが 0 人のとき、追加操作をすると、原点に「メンバー1」が 1 人配置され、
    選択状態になる
  - メンバーが N 人のとき、追加操作をすると、メンバーが N+1 人になり、
    新しいメンバーが 2D エディターと 3D ビューの両方に表示され、選択状態になる

#### メンバー削除

- **入力**: メンバーの選択（1.6）と削除操作、確認ダイアログへの応答
- **正常系の振る舞い**: 削除操作をすると確認ダイアログが表示され、承認すると
  選択中のメンバーが削除される。キャンセルすると削除されない
- **出力**: 削除されたメンバーが 2D エディターと 3D ビューの両方から消える
- **エラー・空状態**: メンバーが未選択のときは削除操作を行えない
- **受け入れ条件**:
  - メンバーを選択して削除操作をすると、確認ダイアログが表示される
  - 確認ダイアログで承認すると、そのメンバーが 2D エディターと 3D ビューの
    両方から消える
  - 確認ダイアログでキャンセルすると、メンバーは削除されない
  - どのメンバーも選択していないとき、削除操作は実行できない

#### 立ち位置変更（ドラッグ・数値入力）

- **入力**: 2D エディター上でのメンバーのドラッグ操作、または選択中メンバー（1.6）の
  X・Y 座標数値入力欄への入力と確定
- **正常系の振る舞い**:
  - ドラッグ: ドラッグに追従してメンバーの立ち位置（座標）が変わる
  - 数値入力: 選択中メンバーの X・Y 座標数値入力欄に、そのメンバーの現在の座標が
    表示される。直接のテキスト入力は、フォーカスを外す、または Enter キーを
    押すと確定する（**仮定:** メンバー名編集（1.5）と同じ確定方法とする）。
    一方、入力欄の増減ボタン（スピナー）のクリック、入力欄にフォーカスがある
    ときの矢印キー、ホイール操作による増減は、操作のたびに即座に確定する
    （**仮定:** 貼り付け（ペースト）は直接のテキスト入力と同じ扱いとし、
    フォーカスを外す・Enter キーで確定する）。いずれの確定方法でも、その値が
    メンバーの立ち位置になる
  - どちらの操作でも、座標は 0.05 単位に丸められる。丸めた結果がステージ領域外に
    なる場合は、ステージ端そのものではなく、領域内に収まる最大/最小の 0.05 単位の
    値に丸められる（端そのものが 0.05 単位の場合は端と一致する。1.6 立ち位置の範囲）
- **出力**: ドラッグ中を含め、3D ビュー上の立ち位置がリアルタイムに追従する（1.6）。
  数値入力の確定操作でも同様に 2D エディター・3D ビューへ反映される
- **エラー・空状態**:
  - **仮定:** 数値入力欄に数値として解釈できない値（空文字を含む）を入力して
    確定した場合、変更を破棄し元の座標を保持する（メンバー名編集の空文字時の
    挙動と一貫させる）
  - メンバーが未選択のとき、数値入力欄は操作できない（無効化される。既存の
    メンバー名編集欄と同様）
- **受け入れ条件**:
  - メンバーを 2D エディター上でドラッグすると、ドラッグ中も 3D ビュー上の
    立ち位置が追従して動く
  - ドラッグを終了すると、その位置がメンバーの立ち位置として保持される
  - ステージの端を越えてドラッグしようとすると、メンバーは領域内に収まる
    最大/最小の 0.05 単位の位置に留まる（端そのものが 0.05 単位の場合は
    端そのものへの配置になる）
  - メンバーを選択すると、数値入力欄にそのメンバーの現在の X・Y 座標が表示される
  - 数値入力欄に値を入力して確定すると、メンバーの立ち位置がその値になり、
    2D エディターと 3D ビューに反映される
  - 数値入力欄の増減ボタン（スピナー）をクリックすると、クリックのたびに
    メンバーの立ち位置が変わり、2D エディターと 3D ビューに反映される
  - 数値入力欄にステージ範囲外の値を入力して確定すると、領域内に収まる
    最大/最小の 0.05 単位の値に丸められる（端そのものが 0.05 単位でない場合は
    端そのものにはならない）
  - 数値入力欄に 0.05 単位でない値を入力して確定すると、最も近い 0.05 単位の
    値に丸められる
  - 数値入力欄に数値として解釈できない値（空文字を含む）を入力して確定すると、
    座標は変更されず元の値のままになる
  - メンバーが未選択のとき、数値入力欄は操作できない（無効化されている）

#### メンバー名編集

- **入力**: メンバーの選択（1.6）と名前の文字列入力
- **正常系の振る舞い**: 入力を確定すると、メンバーの名前が新しい文字列に変わる
- **出力**: 2D エディターと 3D ビューの名前表示が新しい名前になる
- **エラー・空状態**: **仮定:** 空文字で確定した場合は変更を破棄し、元の名前を保つ
- **受け入れ条件**:
  - メンバーを選択して名前を入力・確定すると、2D エディターと 3D ビューの
    名前表示が新しい名前になる
  - 名前を空文字で確定すると、名前は変更されず元のままになる

#### 3D プレビュー

- **入力**: なし（フォーメーションデータの現在の状態を常に表示する）
- **正常系の振る舞い**: 客席中央からステージ全体を見る固定視点で表示する。
  ステージは 1.6 で定めた厚みを持つ直方体として表示し、天面（メンバーが
  立つ面）を客席側の床からその厚み分だけ上げた高さに置く（1.6 ステージ）。
  各メンバーを簡易人物モデルとして、2D エディター上の立ち位置に対応する
  ステージ天面上の位置（1.6 の座標系）に表示する。各モデルには名前を表示する
- **出力**: 3D ビュー上の描画
- **エラー・空状態**: メンバーが 0 人のときは、メンバーのいないステージのみ表示する
- **受け入れ条件**:
  - 3D ビューは、客席中央からステージ全体を見る固定視点で表示される
  - ステージは 1.6 で定めた厚みを持つ直方体として表示され、天面が客席側の
    床より高い位置にある
  - メンバーが存在するとき、各メンバーが 2D エディター上の立ち位置に対応する
    ステージ天面上の位置に簡易人物モデルとして表示される
  - 各人物モデルには、そのメンバーの名前が表示される
  - メンバーが 0 人のとき、3D ビューにはステージのみが表示される

#### JSON エクスポート

- **入力**: エクスポート操作
- **正常系の振る舞い**: 現在のフォーメーションデータ（1.6）を JSON ファイルとして
  ダウンロードする
- **出力**: JSON ファイル
- **エラー・空状態**: メンバー 0 人でもエクスポートできる（空のフォーメーションと
  して出力する）
- **受け入れ条件**:
  - エクスポート操作をすると、現在の全メンバーの名前と立ち位置を含む JSON
    ファイルがダウンロードされる
  - エクスポートした JSON をインポートすると、エクスポート時と同じ
    フォーメーションが再現される

#### JSON インポート

- **入力**: JSON ファイルの選択
- **正常系の振る舞い**: ファイルの内容で現在のフォーメーションを置き換える
- **出力**: 読み込んだフォーメーションが 2D エディターと 3D ビューに反映される
- **エラー・空状態**: 不正なファイル（JSON として解釈できない、または
  フォーメーションデータの形式でない）の場合、エラーダイアログ（2.4）を表示し、
  現在のフォーメーションは変更しない
- **受け入れ条件**:
  - 正しい形式の JSON ファイルを選択すると、現在のフォーメーションがその内容に
    置き換わり、2D エディターと 3D ビューに反映される
  - 不正な形式のファイルを選択すると、エラーダイアログが表示され、現在の
    フォーメーションは変更されない

#### 保存・復元

- **入力**: 保存ボタンの押下
- **正常系の振る舞い**: 保存ボタンを押すと、現在のフォーメーションデータ（1.6）を
  LocalStorage へ保存し、保存完了ダイアログ（2.4）を表示する。アプリを開いたとき、
  保存データがあればそれを復元する
- **出力**: LocalStorage 上の保存データ、起動時の復元されたフォーメーション、
  保存完了ダイアログ
- **エラー・空状態**: 保存データがない場合はメンバー 0 人の空のステージで開始する。
  未保存の変更がある状態でページを閉じる・再読み込みしようとすると、ブラウザの
  確認ダイアログで警告する
- **受け入れ条件**:
  - 保存ボタンを押すと、保存したことを知らせるダイアログが表示される
  - フォーメーションを編集して保存ボタンを押した後にページを再読み込みすると、
    保存した状態が復元される
  - フォーメーションを編集し、保存せずにページを閉じる・再読み込みしようと
    すると、警告（確認ダイアログ）が表示される
  - 未保存の変更がない状態では、警告なしにページを閉じられる
  - 保存データがない状態でアプリを開くと、メンバー 0 人の空のステージが表示される

### 1.6 機能間の共通ルール

- **ステージ**: 幅（x 方向）11.2m × 奥行（y 方向）6m の固定サイズとする
  （MVP ではサイズ変更機能を持たないが、将来変更しやすいことを技術設計で
  考慮する）。3D ビューでは厚み 0.7m の直方体として表示する（1.5 3D プレビュー）
- **座標系**: x 軸・y 軸とも、メートルではなくバミリの通し番号で表す。
  - x 軸（横方向、客席から見て左右）: ステージ中央の番号を 0 とし、右が正。
    番号 1 につき 0.9m（固定）
  - y 軸（奥行方向）: ステージの一番手前（客席に最も近い端）の番号を基準値
    とし、奥へ行くほど基準値から減算した値になる。**仮定:** 基準値は 2、
    番号 1 につき 0.9m（1.6 グリッド表示のグリッド間隔と共通）とする。
    **将来、基準値・増減の向き（加算/減算）・間隔は変更されうるため、技術
    設計ではこれらを差し替え可能な形で定義する**（MVP では設定 UI は作らず、
    コード上の定数として持つ。x 軸の間隔は変更予定がないため定数のままでよい）
  - 2D エディターと 3D ビューは同じ座標系を共有する
- **グリッド表示**: 2D エディター（サムネイル・ポップアップ）のステージには、
  バミリの目安となるグリッド線を、x 軸・y 軸とも上記座標系の番号 1 つごと
  （= 0.9m 間隔）に表示する。ステージ中央（x 軸）・手前端（y 軸）を基準に
  間隔ごとに線を引き、ステージ端に届かない端数分は最後のマスの幅が狭くなる。
  x = 0 の線（センターライン）・y = 0 の線を、他のグリッド線より太い線で
  視覚的に強調する。編集ポップアップでは、
  グリッドの下辺に x 軸の値を、左辺・右辺に y 軸の値を、いずれも上記座標系の
  番号そのものを表示する。グリッドへのスナップは行わない（対象外。1.1）
- **立ち位置の範囲**: メンバーの立ち位置はステージ領域内（端を含む）に制限し、
  0.05 単位に丸める（立ち位置変更・JSON インポート共通のルール）。丸めた結果が
  ステージ領域外になる場合は、端そのものではなく、領域内に収まる最大/最小の
  0.05 単位の値に丸める（端そのものが 0.05 単位の場合は端と一致する）
- **リアルタイム反映**: フォーメーションデータへのあらゆる変更（追加・削除・移動・
  名前変更・インポート・復元）は、即座に 2D 編集ポップアップ・サムネイル・
  3D ビューのすべてへ反映される
- **メンバーの選択**: 2D 編集ポップアップ上でメンバーをクリックする、または
  編集ポップアップのプルダウンでメンバーを選ぶと選択状態になる。選択中の
  メンバーは視覚的に区別される。削除・名前編集は選択中のメンバーが対象となる
- **メンバー数**: 上限は設けない
- **フォーメーションデータ**: メンバーの一覧であり、各メンバーは名前と立ち位置
  （座標）を持つ。JSON エクスポート・インポート・LocalStorage 保存の対象は
  すべてこのデータである（具体的なスキーマは技術設計で定める）
- **対応環境**: PC ブラウザを主対象とし、タブレット・スマートフォンでも表示が
  崩れないレスポンシブなレイアウトとする。タッチ操作への最適化は対象外（1.1）

## 2. 技術設計

### 2.1 システム構成と責務分割

concept.md の技術スタックに従い、バックエンドを持たない SPA として構築する。

**技術スタック（concept.md より）**: React + TypeScript + Vite、
Three.js + React Three Fiber + Drei。状態管理は React 標準機能のみとし、
Zustand は導入しない（MVP の状態規模では不要。必要になった時点で再検討）。

**UI コンポーネントライブラリ**: フォーム系 UI（ボタン・テキスト入力・
ファイル選択）には MUI（`@mui/material` + `@emotion/react` /
`@emotion/styled`）を使う。3D ビュー（Three.js / R3F）と 2D エディターの
俯瞰図（SVG、`FormationStageSvg` とその子要素）は対象外とし、独自描画のまま
とする。2D 編集ポップアップの外枠は MUI の `Dialog`（`DialogContent` /
`DialogActions`）を使う。削除確認・保存完了通知・インポートエラー通知は
2.4 で定めたブラウザ標準ダイアログ（`window.confirm` / `window.alert`）を使う
方針を維持し、MUI の `Dialog` や `Alert` には置き換えない（将来この方針を
変える場合は 2.4 も合わせて更新する）。

**数値入力欄（増減ボタン付き）**: MUI は増減ボタン付きの数値入力コンポーネントを
提供しないため、`@base-ui/react`（Base UI）の `NumberField` を新規依存として
追加する。ネイティブの `<input type="number">` はスピナー操作とテキスト入力を
区別する手段がなく、1.5 立ち位置変更で定めた「テキスト入力は blur/Enter 確定・
スピナー操作は即時反映」という出し分けを実現できないための採用である。Base UI
は非スタイルのヘッドレスコンポーネントのため、MUI 公式ドキュメント（Number
Field ページ）が提供する構成（`FormControl` / `OutlinedInput` / `InputLabel` /
`FormHelperText` と組み合わせる outlined 版のレシピ）を土台に、
`components/parts/NumberField.tsx` として自前の共通部品を作成し、既存の
`TextField` と見た目を揃える。`MemberPositionInput`（後述の「選択中メンバーの
入力欄」の規約）はこれを使う

**開発ツール（グローバル AGENTS.md の規約に従う）**: ESLint / Prettier / Vitest、
E2E テストに Playwright。`.vscode/settings.json` で format on save を有効化する。

**ディレクトリ構成と責務**:

クリーンアーキテクチャに準じたレイヤード構成とする。React 層
（components / hooks / state / App）→ application → domain の一方向の依存とし、
infrastructure は application が定義するインターフェース（ports）を実装する
（依存性逆転）。

```text
src/
├── domain/          # 業務ルール（純粋ロジック）。フォーメーションの型定義、
│                    # メンバー操作、ステージ定数、座標の丸め、
│                    # シリアライズ・バリデーション
├── application/     # ユースケース（保存・復元・インポート・エクスポートの手順）と、
│                    # 永続化先のインターフェース定義（ports）
├── infrastructure/  # ports の実装（LocalStorage 入出力、JSON ファイルの
│                    # 読み込み・ダウンロード）
├── components/      # React コンポーネント。UI としての役割でサブディレクトリに
│   │                # 分類する
│   ├── sections/    # App が配置する画面の区画（3D ビュー、サムネイル、操作 UI）
│   ├── dialogs/     # モーダル・ポップアップ系（2D 編集ポップアップ）
│   └── parts/       # 区画・ダイアログの中で使う部品（2D 俯瞰図の SVG、
│                    # 3D の人物モデル、エラーメッセージ表示など）
├── hooks/           # カスタムフック（ドラッグ操作の座標変換など）
├── state/           # アプリ状態（useReducer + Context）
├── App.tsx          # レイアウト（1.4 の画面構成）と各部の結線
└── main.tsx         # エントリーポイント
```

- `domain`・`application`・`infrastructure` は React に依存しない TypeScript
  のみとする。うち `domain`・`application` はブラウザ API にも依存しない
  純粋なロジックとし、ユニットテストの主対象とする
- ports へ infrastructure の実装を渡す結線（依存性の注入）は、エントリーポイント
  である main.tsx で行う
- 新しいロジックの置き場所は「業務ルール → domain、操作の手順 → application、
  外部 I/O → infrastructure、画面 → components / hooks / state」を基準に判断する
- コンポーネントの参照は「App → `sections`（および `dialogs`）→ `parts`」を
  基本ルールとし、画面全体の骨組み（区画の配置）は App.tsx が担う。
  単一のボタンのような小さな要素を App が直接 `parts` から参照することは許容する
- ステージ寸法・厚み（1.6）は `domain` の定数として一元定義し、2D・3D の
  両方がこれを参照する（将来のサイズ変更に備えた抽象化はしない）
- **座標軸の値とメートルの変換**（1.6 座標系）: x・y は共通の仕組み
  （基準値・1 単位あたりのメートル・増加方向）を `domain` の定数として持ち、
  値 ↔ 基準位置からの符号付き距離（メートル）を相互変換する関数を用意する。
  2D 描画（SVG）・3D 写像・立ち位置のクランプは、すべてこの関数でメートルに
  変換してから幾何計算を行う。x 軸の値（1.6）は今のところ固定だが、y 軸と
  同じ型・関数を使うことで、将来どちらかの値を変える場合も変更箇所が
  1 か所で済むようにする
- **グリッド線とグリッド番号**（1.6 グリッド表示）: グリッド線は各軸の基準
  位置から 1 単位ずつ生成し、ステージ範囲に収まる位置のみを採用する
  （境界に届かない端数分は線を引かず、ステージ枠線が境界を示す）。
  グリッド番号は変換不要で、その線の座標値（バミリ番号）をそのまま表示する。
  生成ロジックは `domain` に置き、2D の SVG 描画（`components/parts`）から
  参照する。SVG はグリッド番号を描画するため、ステージ本体の外側に余白を
  持つ viewBox とする
- 2D の俯瞰図は SVG で描画し、サムネイルとポップアップは同一コンポーネントを
  表示スケールと操作可否の違いで使い分ける
- **選択中メンバーの入力欄**（1.5 立ち位置変更・メンバー名編集）: 選択中メンバーの
  値を編集する入力欄（名前・X 座標・Y 座標。今後増える入力項目も含む）は、
  `components/parts` に 1 フィールド 1 コンポーネントとして置く。各コンポーネントは
  `member: Member | null` を受け取り、未選択時の無効化表示を自身で行う（呼び出し側で
  無効化の分岐を書かない）。入力中の値はコンポーネント内の state として持ち、
  確定時にのみ呼び出し側へ通知し、不正な値の場合は表示を member の現在値へ戻す。
  確定のタイミングは基本的に blur / Enter とするが、`MemberPositionInput`
  （`NumberField` を使う X・Y 座標欄）はスピナー操作・フォーカス中の矢印キー・
  ホイール操作を Base UI の `onValueChange` の `eventDetails.reason` で判別し、
  それらは即座に確定する（1.5 立ち位置変更）。`FormationEditorDialog` はこれらを
  並べて配置するだけとし、今後メンバーの入力項目が増えた場合も同じ形の
  コンポーネントを追加するだけで対応できるようにする（既存の `MemberNameInput`
  もこの形に合わせて `member: Member | null` を受け取るよう変更する）
- テストファイルは対象コードと同階層の `__tests__` ディレクトリに置く
  （例: `domain/coordinates.ts` → `domain/__tests__/coordinates.test.ts`）。
  全レイヤー（`domain` / `application` / `infrastructure` / `components` など）で統一する

### 2.2 データ・状態設計

**ドメイン型**（1.6 のフォーメーションデータに対応）:

```ts
type Member = { id: string; name: string; x: number; y: number };
type Formation = { members: Member[] };
```

- `id` は `crypto.randomUUID()` で生成。名前変更・削除の対象特定に使う
- `x`, `y` は 1.6 の座標系の値そのもの（バミリ番号。メートルではない）。
  基準値・1 単位あたりのメートル・増減の向きは 1.6 の定義に従う
- 実距離（メートル）への変換は 2.1「座標軸の値とメートルの変換」の関数を
  経由する
- メンバー追加時の初期位置（1.5 メンバー追加）は原点（x=0, y=0）とする
- 3D 空間へは、x をメートルに変換した値を X 軸へ、y をメートルに変換した
  「手前端からの距離」を奥行の中心（`STAGE_DEPTH / 2`）基準に反転して
  Z 軸へ写像する（客席側を +Z。手前端が最も +Z 側になる）。ステージ天面は
  Y=`STAGE_HEIGHT`（1.6 ステージの厚み）とし、メンバーは常にこの高さに配置する
- `STAGE_DEPTH / 2` は 3D 配置（ステージ形状・カメラ・上記の写像）でのみ使う
  値であり、1.6 の立ち位置の範囲（メートルへ変換してからステージ内に
  クランプし、値に戻す）とは独立している

**永続化・交換フォーマット**（JSON エクスポート / インポート / LocalStorage 共通）:

```json
{ "version": 1, "members": [{ "id": "...", "name": "...", "x": 0, "y": 0 }] }
```

- `version` は将来のフォーマット変更に備えた識別のみ（変換処理は作らない）
- LocalStorage のキーは `stage-formation-simulator:formation` の 1 つ

**アプリ状態**（`state/` の useReducer + Context で一元管理）:

- `formation: Formation` — 現在のフォーメーション
- `selectedMemberId: string | null` — 選択中メンバー（1.6 メンバーの選択）
- `isEditorOpen: boolean` — 2D 編集ポップアップの開閉
- `isDirty: boolean` — 未保存の変更の有無（保存・復元・起動直後は false、
  フォーメーションへの変更で true）

### 2.3 処理・データフロー

単方向のデータフローとする。

```text
UI 操作（components / hooks）
    ↓ dispatch（追加・削除・移動・名前変更・置換・保存済みマーク など）
reducer が domain のロジックで新しい状態を生成
    ↓ React の再レンダリング
サムネイル・編集ポップアップ（SVG）と 3D ビュー（R3F）が同じ状態を描画
```

- 1.6 の「リアルタイム反映」は、全ビューが同一状態を購読することで実現する。
  ドラッグ中も pointer イベントごとに移動を dispatch する
- ドラッグ座標は SVG のピクセル座標からメートルへ変換し、`domain` の
  `snapPositionWithinStage`（0.05 単位への丸め＋範囲内に収まる 0.05 単位の値への
  丸め。1.6 立ち位置の範囲）でステージ領域内に収める。
  数値入力欄（1.5 立ち位置変更）の確定も、ドラッグと同じ `MOVE_MEMBER` action を
  dispatch し、reducer 経由で `moveMember`（`snapPositionWithinStage` を使う）が
  そのまま行うため、入力経路によらず同じ規則が適用される。新しい action は
  追加しない
- インポート・復元時も同じ `snapPositionWithinStage` を通す（1.6 立ち位置の範囲）
- 保存・復元・インポート・エクスポートは `application` のユースケースとして
  実装し、ports を介して `infrastructure`（LocalStorage・ファイル I/O）を呼ぶ
- インポート: ファイル選択 → テキスト読み込み → `domain` のバリデーション
  （JSON パース＋スキーマ検査）→ 成功時のみフォーメーションを置換
- エクスポート: 現在の状態をシリアライズし、Blob としてダウンロードさせる
  （ファイル名は `formation.json`）
- 保存: シリアライズして LocalStorage へ書き込み、`isDirty` を false にした
  うえで保存完了ダイアログ（2.4）を表示する。起動時に LocalStorage を読み、
  あれば置換する
- 未保存警告（1.5 保存・復元）: `isDirty` が true の間だけ `beforeunload`
  ハンドラーを登録する

### 2.4 エラー処理

- **インポート失敗**（1.5 JSON インポートのエラー系）: バリデーション失敗時は
  状態を変更せず `window.alert` でエラーダイアログを表示する（MVP では独自の
  エラー表示 UI・状態は持たない）
- **保存完了**（1.5 保存・復元）: 保存ボタン押下でのシリアライズ・書き込み後、
  `window.alert` で保存完了ダイアログを表示する
- **LocalStorage の破損データ**: 起動時の読み込みでバリデーションに失敗した場合は
  保存データなしとして扱い、空のステージで開始する（1.5 保存・復元の空状態）
- **削除確認**（1.5 メンバー削除）: `window.confirm` を使う（MVP では独自
  ダイアログを作らない）
- **未保存警告**: ブラウザ標準の `beforeunload` ダイアログを使う

### 2.5 テスト方針

1.5 の受け入れ条件と 1 対 1 で対応させる。検証手段は次の 2 層とし、
テスト名（describe / it）は日本語で書く。

- **ユニットテスト（Vitest + React Testing Library）**: domain のロジックと、
  コンポーネントの振る舞い。TDD の起点としてこちらを先に書く
- **E2E テスト（Playwright）**: ページ再読み込み・ファイルダウンロード・
  ブラウザダイアログ・WebGL 描画など、ユニットテストで検証できない受け入れ条件

| 1.5 の機能 | 先に書くユニットテスト | E2E で担う受け入れ条件 |
| --- | --- | --- |
| 2D エディター表示 | ポップアップの開閉と編集内容の保持、サムネイルの追従、グリッド線とグリッド番号の生成（domain） | — |
| メンバー追加 | 0 人→原点に「メンバー1」、N 人→N+1 人と連番名（domain）、追加したメンバーが選択状態になる（reducer）、両ビューへの表示 | — |
| メンバー削除 | 確認承認で削除・キャンセルで維持（confirm をモック）、未選択時は実行不可 | — |
| 立ち位置変更 | ドラッグでの座標更新と追従、座標値↔メートルの変換関数（x・y 共通の仕組み）、`snapPositionWithinStage`（0.05 単位への丸め、範囲外は範囲内に収まる 0.05 単位の値に丸める。domain）。数値入力欄: 選択中メンバーの現在値表示・確定時の座標反映・不正値時の据え置き・未選択時の無効化・スピナー操作での即時反映（`MemberPositionInput` のコンポーネントテスト） | — |
| メンバー名編集 | 名前の反映、空文字で元に戻る（domain） | — |
| 3D プレビュー | 座標写像関数（domain。天面の高さ Y=`STAGE_HEIGHT` を含む）、0 人時の空表示 | 固定視点での表示、人物モデルと名前ラベルの表示、ステージが厚みのある直方体として見えること（目視） |
| JSON エクスポート | シリアライズ結果、エクスポート→インポートの往復一致（domain） | ファイルのダウンロード |
| JSON インポート | バリデーション（正常・不正）、不正時に状態が変わらないこと、不正時にエラーダイアログを表示すること（`window.alert` をモック） | ファイル選択から反映までの一連の流れ |
| 保存・復元 | isDirty の遷移、シリアライズ・復元ロジック、保存完了ダイアログを表示すること（`window.alert` をモック） | 保存→再読み込みで復元、未保存時の警告、データなしで空ステージ |

- 3D の名前ラベルは Drei の `Html` で DOM として描画し、E2E から名前を
  検証できるようにする
- 3D の見た目（モデルの形状など）の詳細は自動テストせず、目視で確認する

### 2.7 技術的リスク・未決事項

- **React と R3F のバージョン整合**: React Three Fiber / Drei が対応する React・
  Three.js のバージョン組み合わせを構築時に確認して固定する
- **ドラッグ中の再レンダリング性能**: ドラッグごとに全ビューが再描画されるが、
  MVP のメンバー数規模では問題にならない見込み。体感で問題が出た場合のみ
  最適化を検討する（先回りの最適化はしない）
- **CI・テスト環境での WebGL**: jsdom では WebGL を描画できないため 3D の検証は
  E2E に寄せる。Playwright のヘッドレス Chromium はソフトウェアレンダリングで
  WebGL を実行できる想定だが、構築時に動作確認する
- **beforeunload の挙動差**: 警告ダイアログの表示はブラウザ実装に依存する。
  ロジック（isDirty とハンドラー登録）はユニットテストで担保し、実ブラウザでの
  表示は Playwright と目視で確認する
- **端数マスの見た目**: グリッド間隔で割り切れない端（ステージ枠線までの
  半端な幅）に違和感がないか、実装後に目視で確認する
- **保存データの互換性**: y 軸の基準値・増減の向き・間隔（1.6）は将来変更
  されうる定数だが、変更すると LocalStorage の保存データや書き出し済みの
  JSON に含まれる既存の数値の意味が変わってしまう（マイグレーションは
  MVP の対象外。1.1）。将来これらの定数を変更する際は、この互換性の問題を
  別途検討する
